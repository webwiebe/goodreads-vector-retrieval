# Application Architecture

## Overview

This application is a multi-service system that connects a React frontend to a two-agent backend, grounded in a vector database of 13,525 Goodreads books. The architecture is deliberately simple: no orchestration framework, no LangChain abstractions — just typed TypeScript with thin wrappers around Qdrant and OpenRouter.

## Services

### Qdrant (vector database)

Qdrant stores 13,525 book embeddings indexed in an HNSW (Hierarchical Navigable Small World) graph, which allows approximate nearest-neighbour search in O(log n) time.

Each point in the `books` collection has:
- **Vector**: 384-dimensional float32, generated by `BAAI/bge-small-en-v1.5`
- **Payload**: full book metadata (`work_id`, `title`, `author`, `genres`, `description`, `avg_rating`, `ratings_count`, `original_publication_year`, `image_url`)

The payload is returned alongside results so the API never needs a second lookup. Qdrant persists data to a named Docker volume (`qdrant_data`) so the index survives container restarts.

### API (Fastify / Bun)

The API is a Fastify application running on Bun. It owns three concerns:

**1. Routing** — three routes registered under `/api`:
- `POST /chat` — the main agent endpoint
- `GET /books/search` — direct vector search for debugging
- `GET /health` — liveness + ingestion status

**2. Ingestion** — on startup, the API fires off `runIngestion()` as a background task. It checks whether the Qdrant `books` collection already has ≥12,000 points; if so, it skips. If not, it:
1. Extracts `goodreads_works.csv` from the ZIP (only that file, not the 1.3GB reviews file)
2. Parses the CSV with `csv-parse`
3. Builds an embedding string per book: `"{title} by {author}. Genres: {genres}. {description[:500]}"`
4. Batches 32 texts at a time through the embedding model
5. Upserts each batch to Qdrant

This is idempotent: killing and restarting the container resumes from zero (the collection is recreated) or skips entirely (if a previous run completed). A future optimisation would checkpoint progress to Qdrant itself.

**3. Agent orchestration** — delegates to the two-agent pipeline described below.

### Frontend (React + nginx)

The frontend is a static React app served by nginx. nginx also proxies `/api/*` to the API container, so the browser never makes cross-origin requests.

The frontend has one meaningful runtime decision: the **"Use Vector Search"** toggle sets `useRag` in every `POST /api/chat` body. This lets users see the quality difference live, which doubles as an informal evaluation tool.

## Agent pipeline

### Memory Agent (`src/agents/memory-agent.ts`)

The Memory Agent is a thin wrapper that converts a user string into a Qdrant query.

```
user query string
      │
      ▼
  embedText()           ← @huggingface/transformers, BAAI/bge-small-en-v1.5
      │
      ▼  384-dim vector
  searchBooks(vec, topK=10)   ← Qdrant ANN cosine search
      │
      ▼
  filter(score > 0.3)    ← removes weak matches
      │
      ▼
  SearchResult[]         ← { book: Book, score: number }
```

The 0.3 score threshold is intentionally loose. It's better to give the Decision Agent 8 somewhat-relevant books than 2 highly-relevant ones, since the LLM will pick the best 3.

### Decision Agent (`src/agents/decision-agent.ts`)

The Decision Agent takes the retrieved books and the full conversation history, then calls OpenRouter.

System prompt structure:
```
You are a book recommendation expert.

Here are relevant books from the database:
[book 1: title, author, genres, avg_rating, description[:200]]
[book 2: ...]
...

Pick the 3 most suitable books for the user's preferences.
Output ONLY valid JSON:
{
  "recommendations": [{"book_title": "", "book_author": "", "reason": ""}],
  "response": "",
  "follow_up_question": ""
}
```

The model (`qwen/qwen-2.5-7b-instruct`) returns JSON, which is parsed and the `book_title`/`book_author` fields are used to look up the full `Book` object from the retrieved set. This means the API always returns real metadata (real ratings, real cover images) rather than trusting the LLM to reproduce them accurately.

If the LLM hallucinates a book title that doesn't match anything in the retrieved set, that recommendation is silently dropped. This is a deliberate design choice: it's better to return 1 real recommendation than 3 fabricated ones.

### Orchestrator (`src/agents/orchestrator.ts`)

The Orchestrator is the entry point for `POST /api/chat`. It decides which path to take:

```typescript
export async function chat(request: ChatRequest): Promise<ChatResponse> {
  if (request.useRag === false) return callLLMDirect(request);
  const retrieved = await searchMemory(lastUserMessage);
  return recommend(request.messages, retrieved);
}
```

The no-RAG path uses the same LLM with a simpler system prompt and no context books. This path exists purely for evaluation; the responses it produces are used as the baseline in promptfoo comparisons.

## Embedding model

**Model**: `BAAI/bge-small-en-v1.5` (via `@huggingface/transformers`, served by `onnxruntime-node`)

| Property | Value |
|---|---|
| Dimensions | 384 |
| Model size | ~24 MB (quantised to q8) |
| Similarity metric | Cosine |
| Input max tokens | 512 |
| Loaded | Once at startup, cached in memory |
| Cache on disk | `./data/.hf_cache` (persisted via Docker volume) |

The model is loaded once on first embed call and kept in memory for the lifetime of the container. Subsequent calls are fast (~2–5ms per batch of 32).

The same model embeds both the book corpus (at ingestion time) and user queries (at request time). This is essential: if you embed queries with a different model than the corpus, cosine similarity scores become meaningless.

## Data flow: a single chat request

```
1. Browser → POST /api/chat { messages: [{role:"user", content:"I love cozy mysteries"}], useRag: true }

2. Fastify router → orchestrator.chat(request)

3. Orchestrator extracts last user message: "I love cozy mysteries"

4. Memory Agent:
   embedText("I love cozy mysteries")
   → [0.12, -0.45, 0.78, ...]  (384 floats)
   
   searchBooks(vector, topK=10)
   → Qdrant returns top 10 books by cosine similarity
   
   filter(score > 0.3)
   → 8 books pass

5. Decision Agent:
   Builds system prompt with 8 book summaries
   POST https://openrouter.ai/api/v1/chat/completions
     model: qwen/qwen-2.5-7b-instruct
     messages: [system, ...userMessages]
   
   Response (JSON):
   {
     "recommendations": [
       {"book_title": "The Thursday Murder Club", "book_author": "Richard Osman", "reason": "..."},
       {"book_title": "A Fatal Grace", "book_author": "Louise Penny", "reason": "..."},
       {"book_title": "Hamish Macbeth", "book_author": "M.C. Beaton", "reason": "..."}
     ],
     "response": "Here are three cozy mysteries...",
     "follow_up_question": "Do you prefer a rural or urban setting?"
   }

6. Decision Agent maps titles back to full Book objects from retrieved set

7. Orchestrator returns ChatResponse to Fastify

8. Fastify → 200 OK { response, recommendations: [{book, reason}], sources: [...] }

9. React renders BookCard components
```

## Infrastructure

Everything runs as Docker containers orchestrated by Docker Compose.

```yaml
services:
  qdrant:    image: qdrant/qdrant:latest
  api:       build: docker/Dockerfile.api     (oven/bun:1-slim)
  frontend:  build: docker/Dockerfile.frontend (nginx:alpine)
```

The API's Dockerfile uses a two-stage build:
1. **deps stage** (`oven/bun:1-slim`): `bun install --production` — installs only runtime dependencies, runs `onnxruntime-node` postinstall to download platform-correct ONNX binary
2. **runtime stage** (`oven/bun:1-slim`): copies `node_modules` and `src/`, runs `bun src/api/server.ts` directly (no compile step — Bun handles TypeScript natively)

Named volumes:
- `qdrant_data` — Qdrant's index and WAL, survives restarts
- `fastembed_cache` — HuggingFace model cache, avoids re-downloading on restart

Bind mounts:
- `./data` → `/app/data` — extracted CSVs and HF model cache
- `./Goodreads Book Reviews.zip` → `/app/Goodreads Book Reviews.zip` (read-only) — source data

## Evaluation infrastructure

The `evaluation/` directory contains the promptfoo configuration used to compare RAG vs no-RAG quality. It is not part of the Docker Compose setup — it runs locally against the live API.

```
evaluation/
├── promptfoo.yaml     # provider definitions + shared assertions
└── test-cases.yaml    # 20 standardised prompts
```

The two providers (`rag-agent`, `no-rag-agent`) differ only in the `useRag` field sent to `POST /api/chat`. Everything else — the model, the prompt, the response format — is identical. This isolates the retrieval step as the single variable.

See [vector-vs-keyword-search.md](./vector-vs-keyword-search.md) for the full evaluation methodology.

## Extending the system

**Add more data**: the `goodreads_reviews.csv` (1M+ reviews) is present in the ZIP but not indexed. A second collection (`reviews`) with review text and a `work_id` payload could enrich results by matching on review sentiment rather than book metadata.

**Hybrid search**: combine BM25 keyword scoring with vector similarity using Qdrant's built-in sparse vector support. This improves precision for exact-match queries (author names, ISBNs) while retaining semantic recall.

**Re-ranking**: after ANN retrieval, pass results through a cross-encoder re-ranker (e.g., `cross-encoder/ms-marco-MiniLM-L-6-v2`) for higher-precision ordering before handing off to the LLM.

**Multi-turn memory**: the orchestrator currently only uses the last user message for the embedding query. Summarising the full conversation into a single query vector would improve context-aware retrieval.
